System Overview (ภาพรวมของระบบ) Research Rama Tracker (v2.36) เป็น Web Application ที่สร้างขึ้นเพื่อใช้ในการบริหารจัดการและติดตามความก้าวหน้า (Progress Tracking) ของโครงร่างงานวิจัยภายในหน่วยโรคผิวหนัง โรงพยาบาลรามาธิบดี ประโยชน์หลักของระบบ: การติดตามสถานะ: สามารถติดตามขั้นตอนงานวิจัยตั้งแต่ Protocol Draft, การเสนอคณะกรรมการจริยธรรม (EC), การเก็บข้อมูล, จนถึงการตีพิมพ์ (Manuscript Accept) การจัดการสิทธิ์: มีระบบแยกระดับการเข้าถึงข้อมูลและการกระทำระหว่าง ผู้วิจัยหลัก (Main Research - MR) และอาจารย์ที่ปรึกษา (Main Staff - MS) รวมทั้งทีมแพทย์และอาจารย์ร่วมวิจัย ความปลอดภัยอัจฉริยะ: ผสานการทำงานกับการล็อกอินผ่านบัญชี LINE (LIFF) เพื่อยืนยันตัวตนสำหรับกลุ่มอาจารย์ การจัดการเอกสาร: แปลงไฟล์ PDF ของงานวิจัยขึ้นสู่ Google Drive และเก็บเส้นทางลิงก์ (URL) ลง Google Sheets โดยอัตโนมัติ แดชบอร์ด: ระบบสามารถคำนวณวันหมดอายุ EC แจ้งเตือนแบบเรียลไทม์ และสรุปภาระงานรายบุคคลได้ 2. Architecture & Data Flow (สถาปัตยกรรมและการไหลของข้อมูล) สถาปัตยกรรมของระบบเป็นแบบ Serverless โดยพึ่งพาบริการของ Google Workspace เป็นหลัก ทำงานประสานกันดังนี้: Frontend (User Interface): ประกอบด้วย HTML, CSS (Bootstrap), และ JavaScript (SweetAlert2, LIFF) index.html: หน้าจอหลักสำหรับล็อกอิน, สร้าง, และอัปเดตข้อมูลงานวิจัย dashboard2026.html: หน้าแสดงภาพรวมงานวิจัยเรียงตามระดับความคืบหน้า (Score) summary.html: ตารางสรุปภาพรวมจำนวนโปรเจกต์งานวิจัยแบ่งตามแพทย์และอาจารย์แต่ละท่าน (แยกบทบาท Main และ Other) Backend (API Server): เขียนด้วย Google Apps Script (code.gs) ทำหน้าที่รับส่ง Request ผ่านเมธอด doGet และ doPost เพื่อคัดกรองข้อมูลก่อนลงฐานข้อมูลหรืออัปโหลดไฟล์ Database & Storage: Google Sheets (SHEET_ID): เก็บฐานข้อมูล 2 ชีตหลัก ได้แก่ staff (ข้อมูลบุคลากร) และ research69 (ข้อมูลสถานะงานวิจัย) Google Drive (FOLDER_ID): ใช้เก็บไฟล์แนบ PDF ที่ถูกแปลงจาก Base64 Payload Data Flow (การไหลของข้อมูลตอนบันทึก): Frontend แปลงไฟล์ PDF เป็น Base64 → ห่อเป็น JSON Payload ส่งผ่าน POST ไปยัง API → Backend แปลง Base64 กลับเป็น PDF → อัปโหลดขึ้น Drive → นำ URL และข้อมูลลงอัปเดตใน Sheet research69 → รีเทิร์น JSON Status กลับไปที่ Frontend 3. Core Logic & Functions (ลอจิกหลักและการทำงานของโค้ด) Backend (code.gs) getAllResearchData(): ค้นหาบุคลากรใน staff และแยกกลุ่มออกเป็น 2 อาร์เรย์ (doctors ได้แก่ แพทย์ประจำบ้าน/ผู้ช่วยวิจัย และ staffs ได้แก่ อาจารย์) จากนั้นดึงข้อมูลจากชีต research69 เพื่อส่งเป็นก้อนกลับไปให้ Frontend getInitialData(): ใช้ในการยืนยันรหัสบุคลากร ramaID ของผู้ใช้ หากกลุ่มบุคลากรเป็น "อาจารย์" ลอจิกจะบังคับเช็คว่ามี line_uid แนบมาด้วยหรือไม่ และตรงกับฐานข้อมูลเดิมหรือไม่ (Security Validation) ฟังก์ชันนี้ยังส่งรายชื่องานวิจัยกลับไปเฉพาะรายการที่ผู้ใช้เป็น MR, MS, OR, หรือ OS เท่านั้น saveResearchData(): รับ JSON Payload เช็คดูว่ามี Payload File ฐาน 64 (เช่น draftLinkBase64, ecLinkBase64) หรือไม่ หากมีจะสั่งฟังก์ชัน uploadPDF ให้ทำงาน เมื่อเสร็จสิ้นจะเซ็ตค่า URL ลงในอ็อบเจกต์ ก่อนจะเอาไปอัปเดตลงแถวเดิม (rowIndex) หรือแทรกแถวใหม่ (appendRow) Frontend (index.html) Permission Mapping: ใช้ตรวจว่าสิทธิ์ผู้ใช้งานเป็นระดับไหน: isMR (ผู้วิจัยหลัก) / isMS (ที่ปรึกษาหลัก): สามารถแก้ไขข้อมูลและเซ็นชื่อรับทราบสถานะได้ isViewer: หากเป็นผู้ร่วมวิจัยหรืออาจารย์กลุ่มอื่นๆ ระบบจะซ่อนปุ่ม "บันทึก" หรือ "เพิ่มรายชื่อ" (Read-only view) Step Unlocking (renderProgress): ควบคุมการปลดล็อกปุ่มความก้าวหน้า ระบบจะบังคับให้ผู้ใช้บันทึกงานตามลำดับขั้น (ล็อกปุ่มขั้นต่อไปหากขั้นก่อนหน้ายังไม่เคลียร์: !prevOk) 4. Prompt / Input Logic (ลอจิกการรับค่าและตรวจสอบข้อมูล) Rama ID Normalization: ระบบบังคับรับข้อมูล ID แล้วใช้ Regex (replace(/\s+/g, '').toLowerCase().replace(/^0+/, '')) ในการกำจัดช่องว่าง ตัวพิมพ์ และลบเลข 0 นำหน้า เพื่อกันการคีย์ข้อมูลซ้ำซ้อนหรือผิดพลาดเวลาเทียบฐานข้อมูล EC Date Validator: ลอจิกตรวจจับวัน EC หากค่าของ EC_expire_date เทียบกับวันปัจจุบัน ได้ค่าน้อยกว่า 0 จะส่งสัญลักษณ์กระพริบตา (Blink alert) หมดอายุ หากเหลือน้อยกว่า 30 หรือ 90 วันจะมี Tag สีแดงหรือเหลืองตามลำดับ File Selection (PDF Only): กำหนดให้แอตทริบิวต์ฟอร์มรับได้เฉพาะ accept=".pdf" และตรวจสอบว่าถ้าในขั้นตอนนั้นมีเงื่อนไข needsCondition ระบบจะเช็คว่าผู้ใช้แนบไฟล์แล้วหรือยัง (ผ่านตัวแปร pendingFiles) ถ้ายังจะไม่ให้กดปุ่มอัปเดตสเตตัส Mandatory Missing Check: หากในฟอร์มไม่มีการกรอกค่า Title ระบบจะส่งคำเตือนด้วย Swal.fire ป้องกันไม่ให้ส่งข้อมูลขยะเข้าสู่ Database 5. User Guide (คู่มือการใช้งาน) ขั้นตอนการใช้งานสำหรับผู้ใช้ทั่วไป: เข้าสู่ระบบ: กรอกรหัสบุคลากร (Rama ID) เพื่อเข้าสู่ระบบ หากคุณเป็น "อาจารย์" และเป็นการเข้าใช้งานครั้งแรก หรือเข้าผ่านเบราว์เซอร์ใหม่ ระบบอาจเด้งให้คุณล็อกอินและอนุญาตสิทธิ์ผ่านแอป LINE เพื่อยืนยันตัวตน หน้ารวมงานวิจัย: ระบบจะแสดงรายการงานวิจัยของคุณ หากมีไอคอน 🚩 หมายความว่ามีสเตตัสที่รอให้คุณดำเนินการอัปเดต หากต้องการริเริ่มงานใหม่ ให้กด + สร้างงานวิจัยใหม่ กรอกข้อมูลหลัก: ใส่ชื่อเรื่อง (Title), เลือกรายชื่อแพทย์ที่ร่วมวิจัยด้วย, และเลือกอาจารย์ที่ปรึกษาหลักและอาจารย์ร่วม แนบไฟล์เริ่มต้น: หากเพิ่งร่าง Protocol เสร็จ ให้ทำการแนบไฟล์ PDF ต้นฉบับในกล่องอัปโหลดด้านล่างส่วนข้อมูลทั่วไป อัปเดตความก้าวหน้า (Progress): เลื่อนมาที่กล่อง "ความก้าวหน้า" หากสเตปไหนคุณรับผิดชอบ ปุ่มจะเป็นสีฟ้าขอบน้ำเงิน ให้คลิกอัปเดต (ระบบจะประทับเวลาให้อัตโนมัติ) ในบางขั้นตอน (เช่น EC Approve) คุณต้องกรอก วันที่อนุมัติ, วันหมดอายุ, หรือ แนบไฟล์ PDF ก่อน ปุ่มบันทึกถึงจะสามารถกดได้ บันทึกข้อมูล: กดปุ่มเขียว "💾 บันทึกข้อมูล" ที่ด้านล่างสุด กรุณารอระบบทำการอัปโหลดไฟล์สู่ฐานข้อมูล ดู Dashboard & Summary: คุณสามารถคลิกปุ่ม "📊 Dashboard" ที่มุมขวาบน หรือ "📈 ดูสรุปภาระงานวิจัย (Summary)" ในหน้าแรก เพื่อดูสรุปความคืบหน้าของคนทั้งหน่วยได้ 6. 1. System Overview (ภาพรวมของระบบ)

Web Application Research Rama Tracker พัฒนาขึ้นเพื่อเพิ่มประสิทธิภาพในการบริหารจัดการโครงร่างงานวิจัยและติดตามความก้าวหน้า (Progress Tracking) ของบุคลากรภายในหน่วยโรคผิวหนัง คณะแพทยศาสตร์โรงพยาบาลรามาธิบดี

ประโยชน์และฟังก์ชันเด่นของระบบ:

  • Real-time Tracking: ติดตามสเตตัสงานวิจัยได้อย่างละเอียดถึง 19 ขั้นตอนย่อย ตั้งแต่การยกร่าง Protocol ไปจนถึงขั้น Manuscriptได้รับการตอบรับ (Accept)
  • Role-Based Access Control (RBAC): แยกระดับสิทธิ์และหน้าที่ชัดเจนระหว่าง ผู้วิจัยหลัก (Main Research), อาจารย์ที่ปรึกษาหลัก (Main Staff) และผู้ร่วมวิจัย (Other)
  • LINE Login Integration (LIFF): เสริมความปลอดภัยขั้นสูงสำหรับกลุ่มอาจารย์แพทย์ ป้องกันการเข้าถึงข้อมูลและการอนุมัติโดยไม่ได้รับอนุญาต
  • Automated Cloud Storage: ระบบแปลงเอกสาร PDF เป็น Base64 อัปโหลดขึ้น Google Drive ของหน่วยงานโดยอัตโนมัติ และบันทึก URL ลิงก์กลับลงฐานข้อมูลอย่างเป็นระบบ

2. Architecture & Data Flow (สถาปัตยกรรมและการไหลของข้อมูล)

ระบบถูกออกแบบบนโครงสร้างแบบ Serverless Architecture โดยทำงานร่วมกันระหว่าง 3 ส่วนหลัก ดังนี้:

Frontend

HTML5, CSS3 (Bootstrap 5), JavaScript (LIFF SDK, SweetAlert2)

Backend (API)

Google Apps Script (code.gs) ประมวลผลผ่านฟังก์ชัน doGet และ doPost

Database & Storage

Google Sheets (ชีต staff และ research69) และ Google Drive Folder

แผนผังการไหลของข้อมูล (Data Flow) ในการอัปเดตข้อมูลและไฟล์:
  1. ผู้ใช้แนบไฟล์ PDF และกรอกข้อมูลบนฟอร์มหน้าเว็บ (Frontend)
  2. JavaScript แปลงไฟล์เป็น Base64 DataURL และจัดโครงสร้างในรูปแบบ JSON Payload
  3. ส่งข้อมูลผ่าน HTTP POST Request ไปยังตัวประมวลผลของ Google Apps Script (Backend)
  4. สคริปต์เซิร์ฟเวอร์ถอดรหัส Base64 สั่งสร้างไฟล์ PDF บน Google Drive ปรับสิทธิ์เป็น "ทุกคนที่มีลิงก์สามารถดูได้" และส่งค่า URL กลับมา
  5. สคริปต์เขียนข้อมูลข้อความ พร้อมลิงก์ URL ลงในแถวข้อมูลของชีต research69 ตามดัชนี rowIndex ที่ถูกต้อง

3. Core Logic & Functions (ลอจิกหลักของโค้ด)

สคริปต์ฝั่งฝั่งเซิร์ฟเวอร์ (code.gs)
ชื่อฟังก์ชัน ลอจิกและการทำงานภายใน
doGet(e) เราเตอร์ฝั่งรับข้อมูลเข้าผ่าน URL พารามิเตอร์ คัดกรอง Action หากเป็น getInitialData จะสั่งรันระบบล็อกอินและดึงงานวิจัยเฉพาะของผู้ใช้ หากเป็น getAllResearch จะส่งคืนข้อมูลงานวิจัยทั้งหมดไปประมวลผลหน้าแดชบอร์ด
getInitialData() ตรวจสอบรหัส Rama ID หากผู้ใช้เป็นกลุ่ม "อาจารย์" จะตรวจเช็ค line_uid เพื่อทำระบบ LINE Login สำหรับยืนยันตัวตน และคัดกรองส่งงานวิจัยคืนไปเฉพาะชิ้นที่ผู้ใช้มีส่วนเกี่ยวข้อง (เป็น MR, MS, OR, OS)
saveResearchData() ตรวจสอบ JSON Payload ที่ส่งมาจากหน้าบ้าน ตรวจเช็คก้อน Base64 ของไฟล์ในแต่ละขั้นตอน หากพบข้อมูลจะสั่งอัปโหลดไฟล์ผ่าน uploadPDF() ก่อนจะนำอาร์เรย์ผลลัพธ์แมปลงคอลัมน์ใน Google Sheets ตามชื่อหัวตาราง (Headers Matching)
uploadPDF() ตัดแยก Header ของ Base64 เพื่อหา Content-Type ถอดรหัสด้วย Utilities.base64Decode() แล้วบันทึกลงโฟลเดอร์ปลายทาง พร้อมตั้งสิทธิ์เป็นแบบ View-only (Anyone with link)
สคริปต์ฝั่งหน้าเว็บ (index.html)
  • ระบบตรวจสอบลำดับขั้นตอน (Step Unlocking): ฟังก์ชัน renderProgress() ควบคุมการปิดกั้น/เปิดปุ่มแสตมป์เวลา โดยจะเช็คค่า prevOk หากขั้นตอนก่อนหน้ายังไม่มีข้อมูลแสตมป์เวลา ปุ่มในขั้นตอนถัดไปจะถูกเซ็ตสถานะเป็น disabled และเปลี่ยนคลาสเป็น step-locked ทันที
  • สิทธิ์การแสดงผล (Permission Mapping): ระบบจะคำนวณสิทธิ์ผู้ใช้งาน หากผู้ใช้ปัจจุบันไม่ได้เป็นผู้วิจัยหลัก (isMR) และไม่ได้เป็นอาจารย์ที่ปรึกษาหลัก (isMS) ตัวแปร isViewer จะมีค่าเป็น True ส่งผลให้ซ่อนปุ่มบันทึก ปุ่มเพิ่มรายชื่อ และซ่อนอินพุตสำหรับอัปโหลดไฟล์ทันที

4. Prompt & Input Logic (การตรวจสอบและจัดการข้อมูลขาเข้า)

ลอจิกการเคลียร์ข้อมูลไอดี (Rama ID Normalization)

เพื่อป้องกันความผิดพลาดจากการพิมพ์ช่องว่าง หรือการพิมพ์เลขศูนย์นำหน้าไม่เท่ากัน ระบบใช้ฟังก์ชัน normalizeID(id) ในการแปลงค่าข้อมูลก่อนนำไปเปรียบเทียบในฐานข้อมูลด้วยคำสั่ง:

return id.toString().replace(/\s+/g, '').toLowerCase().replace(/^0+/, '') || "0";
ลอจิกแจ้งเตือนวันหมดอายุใบอนุมัติจริยธรรม (EC Expire Date Analytics)

ระบบคำนวณระยะต่างระหว่างวันปัจจุบันกับวันหมดอายุของ EC (EC_expire_date) และส่งคืน Tag Badge เตือนภัยบนการ์ดงานวิจัยดังนี้:

  • น้อยกว่า 0 วัน (หมดอายุแล้ว): แสดง Tag สีแดงพร้อมเอฟเฟกต์ไฟกะพริบแจ้งเตือน (Blink Alert)
  • เหลือไม่เกิน 30 วัน: แสดง Tag สีแดง แจ้งเตือนระดับวิกฤต
  • เหลือไม่เกิน 90 วัน: แสดง Tag สีเหลือง แจ้งเตือนให้เตรียมตัวต่ออายุ
  • มากกว่า 90 วัน: แสดง Tag สีเขียว สถานะปกติความปลอดภัยสูง

5. User Guide (ขั้นตอนการใช้งานสำหรับบุคลากร)

1 เข้าสู่ระบบ (Authentication)

กรอกรหัสบุคลากร (Rama ID) แล้วกดปุ่มเข้าสู่ระบบ สำหรับกลุ่มอาจารย์แพทย์หากเข้าใช้งานครั้งแรก ระบบจะแสดงป๊อปอัปแจ้งเตือนให้กดสิทธิ์ล็อกอินผ่าน LINE เพื่อทำการผูกบัญชีความปลอดภัย

2 เลือกหรือสร้างงานวิจัย

คลิกที่แถบเลือกงานวิจัย ระบบจะเรียงลำดับงานที่มีเครื่องหมายธงสีแดง 🚩 ไว้ด้านบนสุด ซึ่งหมายถึง "เป็นงานที่คุณต้องเข้าไปอัปเดตความคืบหน้าในปัจจุบัน" หรือคลิกเลือก + สร้างงานวิจัยใหม่ หากต้องการเปิดเรื่องใหม่

3 จัดการข้อมูลทีมวิจัย

กรอกชื่อเรื่อง (Title) จากนั้นสามารถกดปุ่ม + เพิ่มแพทย์ หรือ + เพิ่มอาจารย์ เพื่อเลือกรายชื่อทีมผู้ร่วมวิจัยจากหน้าต่างรายชื่อป๊อปอัปได้อย่างสะดวก

4 การแสตมป์เวลาและแนบเอกสาร

ในสเต็ปงานที่คุณมีสิทธิ์ ปุ่มกดจะเป็นคำว่า ลงเวลา หรือ 🔄 อัปเดตข้อมูล ให้ทำการแนบไฟล์ PDF ประจำขั้นตอนนั้นๆ (ถ้ามีกำหนดไว้ เช่น ขั้นตอนอนุมัติ EC หรือร่างแมนูสคริปต์) จากนั้นกดปุ่มแสตมป์ลงเวลา ระบบจะประทับเวลาปัจจุบันให้ทันที

5 บันทึกข้อมูลขึ้นระบบคลาวด์

ตรวจสอบข้อมูลให้แน่ใจแล้วกดปุ่มสีเขียว 💾 บันทึกข้อมูล ด้านล่างสุด ระบบจะซิงก์ข้อมูลอัปโหลดไฟล์ขึ้น Google Drive และบันทึกประวัติลง Google Sheets จนเสร็จสิ้น

6. Troubleshooting & Maintenance (การแก้ไขปัญหาและการดูแลรักษา)

สาเหตุ: เกิดจากอาจารย์แพทย์ท่านนั้นๆ มีการเปลี่ยนสลับบัญชี LINE หรือเคยใช้ LINE Account อื่นล็อกอินค้างไว้ในระบบเป็นครั้งแรก
วิธีแก้ไขสำหรับผู้ดูแลระบบ (Admin): ให้เปิดไฟล์ฐานข้อมูล Google Sheets เข้าไปที่ชีตชื่อ staff ค้นหาแถวรายชื่อของอาจารย์ท่านนั้น แล้วทำการลบข้อมูลในช่องคอลัมน์ line_uid และ line_displayname ให้เป็นช่องว่างว่างๆ จากนั้นแจ้งให้อาารย์ล็อกอินใหม่อีกครั้งเพื่อผูกไอดีใหม่

สาเหตุ: เพื่อความปลอดภัยของข้อมูล ระบบมีเงื่อนไขล็อกไม่ให้แพทย์แก้ไขเปลี่ยนตัวอาจารย์ที่ปรึกษาหลัก หากอาจารย์ที่ปรึกษาท่านนั้นๆ ได้เริ่มทำการกดอนุมัติหรือแสตมป์เวลาความก้าวหน้าในขั้นตอนฝั่งของอาจารย์ (MS Action) ไปแล้ว
วิธีแก้ไข: หากมีความจำเป็นเร่งด่วนในการเปลี่ยนย้ายตัวอาจารย์ที่ปรึกษาหลัก แอดมินผู้ดูแลระบบจะต้องเข้าไปทำการแก้ไขรหัสและชื่ออาจารย์ด้วยตนเองโดยตรงในแถวข้อมูลของชีต research69

สาเหตุ: ไฟล์เอกสาร PDF มีขนาดใหญ่เกินกว่าขีดจำกัดที่ก้อนข้อมูล String ของ Google Apps Script API จะรับได้ในหนึ่งตารางเวลา หรือโฟลเดอร์ Google Drive ปลายทางถูกจำกัดสิทธิ์เขียนไฟล์
วิธีแก้ไข: ให้ทำการบีบอัดขนาดไฟล์ PDF (Compress PDF) ให้เล็กลงก่อนทำการอัปโหลด (แนะนำไม่ควรเกิน 5-10 MB) และตรวจสอบว่าตัวแปร FOLDER_ID ในไฟล์ code.gs ยังสอดคล้องกับโฟลเดอร์คลาวด์ที่มีสิทธิ์เขียนแชร์ข้อมูลอยู่จริง
การเพิ่มรายชื่อบุคลากรใหม่เข้าระบบ (Maintenance)

ผู้ดูแลระบบไม่จำเป็นต้องเข้ามาแก้ไขโค้ดโปรแกรมแต่อย่างใด เพียงแค่เปิด Google Sheets ชีต staff แล้วทำการกรอกข้อมูลพนักงานใหม่ลงไปในแถวถัดไป โดยระบุ ramaID, name และช่อง group ให้มีคำระบุประเภทชัดเจน (เช่นมีคำว่า "อาจารย์" หรือกลุ่ม "แพทย์ประจำบ้าน") ระบบหน้าบ้านจะทำการดึงข้อมูลไปแสดงผลในตัวเลือกและคัดแยกสิทธิ์ให้โดยอัตโนมัติในการล็อกอินครั้งถัดไป
